# Schema Required Fields Issue Analysis

## Problem Statement

The generated Zod schemas mark **all fields as optional** (`.optional()`), even when Google's API documentation clearly states they are "Required". This creates several issues:

1. **Runtime validation doesn't catch missing required fields**
2. **TypeScript types allow undefined for required fields**
3. **AI agents don't know which fields are mandatory**
4. **API calls fail with unclear errors when required fields are missing**

## Example

```typescript
// Current generated schema (WRONG)
export const InsertionOrder = z.object({
  /** Required. The display name of the insertion order. Must be UTF-8 encoded with a  */
  displayName: z.string().optional(), // ❌ Should NOT be optional

  /** Required. Immutable. The unique ID of the campaign that the insertion order belongs to */
  campaignId: z.string().optional(), // ❌ Should NOT be optional

  /** Required. The budget allocation settings of the insertion order. */
  budget: z.lazy(() => InsertionOrderBudget).optional(), // ❌ Should NOT be optional
});
```

## Root Cause

### Google Discovery Document Format

Google's Discovery Document (the source for our schema generation) **does not include a `required` field array**. Instead, "Required" is only indicated in field descriptions as human-readable text.

**Verification:**

```bash
curl -s 'https://displayvideo.googleapis.com/$discovery/rest?version=v4' \
  | jq '.schemas.InsertionOrder.required'
# Output: null
```

The InsertionOrder schema structure:

```json
{
  "type": "object",
  "properties": {
    "displayName": {
      "type": "string",
      "description": "Required. The display name of the insertion order. Must be UTF-8 encoded with a maximum size of 240 bytes."
    },
    "campaignId": {
      "type": "string",
      "description": "Required. Immutable. The unique ID of the campaign that the insertion order belongs to."
    }
  }
  // Note: NO "required" array!
}
```

### Schema Generation Pipeline

Our schema generation pipeline:

1. **Fetch Discovery Doc** → Contains descriptions with "Required" text
2. **Convert to OpenAPI** (`convert-to-openapi.ts:152-155`) → Preserves `required` array IF present
3. **Generate Zod** (`generate-zod-schemas.ts:152-159`) → Marks fields optional unless in `required` array

```typescript
// generate-zod-schemas.ts:152-159
const isRequired = schema.required?.includes(propName);

if (!isRequired) {
  propLine += ".optional()"; // ← Everything becomes optional!
}
```

Since Discovery Doc has no `required` array, ALL fields become optional.

```typescript
  // Missing displayName, campaignId, budget - but Zod says it's valid!

```

Expected behavior:

```typescript
InsertionOrder.parse(invalidIO); // ❌ Should throw ZodError
```

### 2. **TypeScript Types**

Current types (generated by openapi-typescript):

```typescript

  displayName?: string; // ❌ Should be required
  campaignId?: string; // ❌ Should be required
  budget?: InsertionOrderBudget; // ❌ Should be required
```


### 3. **AI Agent Guidance**

Without schema-level required field enforcement:

- AI doesn't know `campaignId` is mandatory in data payload
- Relies only on our custom relationship system (which we just built)

### 4. **API Call Failures**

When required fields are missing, the DV360 API returns:

```json
{
  "error": {
    "code": 400,
    "message": "Request contains an invalid argument.",
    "status": "INVALID_ARGUMENT"
  }
}
```

This is **not helpful** - doesn't tell us which field is missing!

## Solutions

### Option 1: Parse "Required" from Descriptions (Recommended)

Modify `convert-to-openapi.ts` to extract "Required" from descriptions:

```typescript
function extractRequiredFields(schema: DiscoverySchema): string[] {
  const required: string[] = [];

  if (schema.properties) {
    for (const [propName, propSchema] of Object.entries(schema.properties)) {
      if (isRequiredFromDescription(propSchema.description)) {
        required.push(propName);
      }
    }
  }

  return required;
}

function isRequiredFromDescription(description?: string): boolean {
  if (!description) return false;

  // Match various patterns:
  // - "Required. ..."
  // - "Required: ..."
  // - Starts with "Required"
  const requiredPattern = /^Required[\.:]/i;
  return requiredPattern.test(description.trim());
}

// Then in convertSchema:
if (discoverySchema.properties) {
  // ... convert properties ...

  // Extract required fields from descriptions
  const requiredFields = extractRequiredFields(discoverySchema);
  if (requiredFields.length > 0) {
    openApiSchema.required = requiredFields;
  }
}
```

**Pros:**

- ✅ Uses the information Google provides
- ✅ Automatic for all entities
- ✅ No manual maintenance needed
- ✅ Works with existing Zod generator

**Cons:**

- ⚠️ Relies on text parsing (could have false positives/negatives)
- ⚠️ Google could change description format
- ⚠️ Doesn't distinguish between required-on-create vs required-on-update

### Option 2: Manual Required Field Registry

Create a manual registry of required fields:

```typescript
const REQUIRED_FIELDS: Record<string, string[]> = {
  InsertionOrder: [
    "displayName",
    "campaignId",
    "advertiserId",
    "entityStatus",
    "budget",
    "pacing",
    "frequencyCap",
    "kpi",
  ],
  Campaign: ["displayName", "advertiserId", "campaignGoal"],
  LineItem: [
    "displayName",
    "advertiserId",
    "insertionOrderId",
    "lineItemType",
    "entityStatus",
    "flight",
    "budget",
    "pacing",
    "frequencyCap",
  ],
  // ... for all 12 entity types
};
```

**Pros:**

- ✅ 100% accurate
- ✅ Can handle special cases
- ✅ Can distinguish create vs update requirements

**Cons:**

- ❌ Manual maintenance required
- ❌ Must update when API changes
- ❌ Duplication of Google's documentation
- ❌ Doesn't scale to new entities

### Option 3: Hybrid Approach

Combine both approaches:

1. **Auto-detect from descriptions** (Option 1)
2. **Override with manual registry** for special cases
3. **Validation warnings** if description says "Required" but field not in manual list

```typescript
function getRequiredFields(schemaName: string, schema: DiscoverySchema): string[] {
  // Start with auto-detected
  const autoDetected = extractRequiredFields(schema);

  // Override with manual if present
  const manual = REQUIRED_FIELDS_OVERRIDES[schemaName];
  if (manual) {
    return manual;
  }

  return autoDetected;
}
```

### Option 4: Do Nothing (Current State)

Rely entirely on our custom relationship validation system.

**Pros:**

- ✅ Already implemented
- ✅ Handles parent relationships correctly
- ✅ No schema generation changes needed

**Cons:**

- ❌ Doesn't validate non-relationship required fields (e.g., `displayName`, `budget`)
- ❌ No TypeScript compile-time safety
- ❌ Validation only at create/update time, not at data construction time

## Recommendation

**Implement Option 1 (Parse from descriptions) with Option 3 enhancements (manual overrides)**

### Implementation Plan

1. **Add description parsing to `convert-to-openapi.ts`**
   - Extract "Required" from field descriptions
   - Add to OpenAPI schema `required` array
   - Log warnings for ambiguous cases

2. **Add manual override registry**
   - Create `REQUIRED_FIELDS_OVERRIDES` for special cases
   - Use for fields that are:
     - Required only on create (not update)
     - Required conditionally
     - Incorrectly detected from description

3. **Update Zod generation** (no changes needed - already checks `required`)

4. **Regenerate schemas** with new pipeline

5. **Test validation**
   - Verify required fields throw errors
   - Verify optional fields allow undefined
   - Test with actual API calls

## Additional Considerations

### Output-Only Fields

Some fields are marked "Output only" (e.g., `insertionOrderId`, `updateTime`). These should be:

- **Optional in create schemas** (not provided by user)
- **Required in response schemas** (always returned by API)

Current generator doesn't distinguish between request/response schemas.

### Immutable Fields

Fields marked "Immutable" (e.g., `campaignId`) should be:

- **Required on create**
- **Not allowed on update** (or ignored)

Could add Zod refinements:

```typescript
.refine((data) => {
  // On update, campaignId must match existing or be omitted
}, { message: "campaignId is immutable and cannot be changed" })
```

### Conditional Requirements

Some fields are required only in certain conditions:

- "Required when X is Y"
- "One of A or B must be present"

These need custom Zod refinements that can't be auto-generated.

## Conclusion

The root cause is **Google's Discovery Document lacks structured `required` field information**. Our schema generator correctly processes what it receives, but the source data is incomplete.

The recommended solution is to **extract required field information from description text**, with manual overrides for edge cases. This provides:

- Automatic required field detection
- Better runtime validation
- Improved TypeScript types
- Clearer error messages for AI agents
- Protection against invalid API calls

This complements our existing relationship validation system by adding a second layer of defense.

# Connector-side Governance: Decision Tokens + Full Write-Contract Coverage (§10 Item 5)

**Status:** Design approved (brainstorming complete) — 2026-06-01
**Repo:** `cesteral-mcp-servers` (the `mcp-open-advertising` connector fleet)
**Companion repo:** `cesteral-governance-layer` / `cesteral-intelligence` (mints tokens, admits contracts)

---

## 1. Problem

Governance §10 item 5 is out-of-repo for the connector: expand connector contracts/snapshots so **every production write** has a stable contract, preview (dry-run) semantics, execute snapshots, and a reconciliation path — and have the connector **verify the `X-Cesteral-Decision-Token`** that Intelligence now mints on write calls.

Today only the 13 `*_update_entity` tools (+ amazon-dsp `update_commitment`) carry `cesteral` annotations. ~109 other write tools (create/delete/bulk/duplicate/adjust-bids/upload/report-schedule CRUD/conversion-upload/graphql-bulk) are ungoverned. Nothing reads `X-Cesteral-Decision-Token`.

**Scope:** all 122 production write tools across 13 servers (Scope C), plus the net-new token-verification subsystem. Multi-PR, **warn-only first**; enforce per-contract only when coverage is complete.

### Token claims (minted by Intelligence)

| Claim               | Value / meaning                                                           |
| ------------------- | ------------------------------------------------------------------------- |
| `iss`               | `cesteral-intelligence`                                                   |
| `aud`               | `mcp-open-advertising`                                                    |
| `sub`               | tenant                                                                    |
| `contractId`        | governed contract id, e.g. `meta.update_entity.v1`                        |
| `definitionHash`    | SHA-256 of the connector tool definition (from `@cesteral/contract-hash`) |
| `actionHash`        | `sha256(stableStringify(executable write args))`                          |
| `approvalId`        | optional                                                                  |
| `iat`, `exp`, `jti` | standard; `jti` consumed exactly once                                     |

Connector must: verify signature/claims, bind the token to the actual write (`contractId` + `definitionHash` + `actionHash`), consume `jti` exactly once, support warn-only rollout before enforce.

## 2. Architecture — three layers

Matches the repo's existing architecture (`computeDefinitionHash` centralized in `packages/contract-hash/src/index.ts`; all validated tool calls funnel through `packages/shared/src/utils/tool-handler-factory.ts`).

1. **Canonicalization layer — `@cesteral/contract-hash`** — single cross-repo source of truth for canonical bytes. Adds `stableStringify`, `hashActionInput`, `canonicalizeExecutableArgs`. Governance imports this package instead of keeping its own copy → drift is structurally impossible, not convention-bound.
2. **Enforcement layer — `@cesteral/shared`** — decision-token verifier + pluggable `JtiStore` + three-tier mode config + audit/telemetry, invoked from `tool-handler-factory`, gated to `cesteral.kind === "write"`.
3. **Contract-surface layer — annotations + structured responses** — `writeClass: "entity" | "effect"` schema migration, then expansion to all 122 write tools.

**Key lever:** the security layer is independent of snapshot semantics. A token binds `contractId + definitionHash + actionHash` whether or not the write produces an entity snapshot — so uploads/report-schedules get full token verification without a canonical snapshot.

## 3. Canonicalization layer (`@cesteral/contract-hash`)

New exports alongside existing `computeDefinitionHash`:

```ts
// JSON-compatible only. THROWS (does not silently coerce) on:
//   BigInt, Date, NaN, Infinity, function, symbol, class instance, undefined-at-root.
// Reuses the existing sortKeysDeep so definitionHash and actionHash share one canonicalizer.
export function stableStringify(value: unknown): string;

// sha256(stableStringify(value)) -> lowercase hex, no prefix
export function hashActionInput(value: unknown): string;

// Produces the JSON-safe value that BOTH repos hash. See §5 "executable args".
export function canonicalizeExecutableArgs(opts: {
  rawArgs: unknown; // wire JSON-RPC `arguments`
  exclude: string[]; // control fields, e.g. ["dry_run"]
}): unknown;
```

### Golden vectors

`tests/golden-action-hashes.json` — `{ input, stableString, actionHash }` triples, ideally generated by governance-layer's real `stableStringify`. Until governance vectors land, generated locally and marked `source: "connector-pending-governance-confirmation"` so CI surfaces them as unverified.

**Edge cases pinned:** dropped `undefined` properties, `null`, `-0`, nested-array element order, unicode/non-ASCII, large integers, empty objects/arrays. These are exactly where two "identical-looking" stringifiers silently diverge.

## 4. Enforcement layer (`@cesteral/shared`)

### Modules

- `src/governance/decision-token.ts` — `verifyDecisionToken(opts): DecisionTokenVerdict`
- `src/governance/jti-store.ts` — `JtiStore` interface + `InMemoryJtiStore` + `FirestoreJtiStore`
- `src/governance/config.ts` — three-tier mode resolution
- `src/governance/audit.ts` — structured verdict logging + metrics

### Token hardening (built in from PR 1)

- **`alg` allowlist: HS256 only.** Reject `alg: "none"` and any non-HS256 before signature work.
- Optional **`kid`** header honored.
- **Key rotation:** verify against `GOVERNANCE_DECISION_TOKEN_SECRET`, then `GOVERNANCE_DECISION_TOKEN_SECRET_PREVIOUS` (current + previous).

### Verification order (in `tool-handler-factory`)

Runs after `tool.inputSchema.parse(args)` and **after advertiser-scope authz**, before `tool.logic(...)`:

1. Resolve mode for `contractId` (per-contract → per-server → global; global default `off`).
2. `off` → skip (read-only behavior preserved).
3. Read `X-Cesteral-Decision-Token` from `RequestContext.decisionToken` (typed field; see §6).
4. Verify in order, each with a distinct `reasonCode`:
   - `MISSING_TOKEN`
   - `MALFORMED_TOKEN` (not 3 segments / undecodable header / claim present but wrong type)
   - `UNSUPPORTED_ALG` (header `alg !== "HS256"`)
   - `SECRET_UNCONFIGURED` (no non-empty secret configured — fail closed; an empty/whitespace secret is never used as an HS256 key)
   - `INVALID_SIGNATURE` (incl. unknown/mismatched `kid`; `kid` pins its secret with no fallback)
   - `MISSING_CLAIM` — **any** of the required claims absent: `jti`, `exp`, `iat`, `sub`, `contractId`, `definitionHash`, `actionHash`. A signed token missing `jti`/`exp` must NEVER reach `OK` or `consumeOnce`.
   - `EXPIRED` (small clock-skew leeway)
   - `WRONG_ISSUER` / `WRONG_AUDIENCE`
   - `CONTRACT_MISMATCH` (`contractId` ≠ tool annotation `contractId`)
   - `DEFINITION_HASH_MISMATCH` (≠ resolved manifest hash). The expected hash is **optional**: when the caller has no manifest resolver, this check is skipped and the verdict reports `definitionHashVerified: false` — every OTHER binding still runs. The factory fails closed under `enforce` when the hash is unresolved (never silently skips the whole verifier).
   - `ACTION_HASH_MISMATCH` (≠ `hashActionInput(canonicalizeExecutableArgs(...))`)
   - `REPLAYED_JTI` (**consume last**, only after all above pass)
   - else `OK`
5. **`jti` consumed only on an otherwise-valid token** — a missing-claim / malformed / expired / mismatched / unauthorized call never burns a legitimate `jti`. Signature is verified with `jose.compactVerify` (signature only), so the connector owns claim-check ordering; required-claim presence+type is an explicit step before any binding check. **Replay TTL = `max(jtiTtlMs, token-remaining-lifetime)`** so a short `jtiTtlMs` can never let an un-expired token replay after the store entry lapses.
6. Apply mode:
   - `warn` → log verdict (incl. failures), **proceed** with write.
   - `enforce` → any non-`OK` → throw `McpError(JsonRpcErrorCode.Unauthorized)` with reason code; write never runs (surfaced as an `isError` tool result carrying JSON-RPC `-32006`).
7. Forward `jti` (preferred) or `actionHash` as upstream **idempotency key** where the platform API supports one, via `sdkContext.idempotencyKey`.

**Deployment guard:** if any governed write resolves to `enforce` but no distributed `JtiStore` is injected, the factory logs a one-time registration WARNING (in-memory replay protection is per-instance only).

### Ordering vs advertiser-scope authz

Scope authz runs **before** token verification/jti consumption, so unauthorized calls are rejected without burning a valid `jti`.

## 5. "Executable args" — the critical cross-repo contract

The highest-risk item. `actionHash` must be byte-identical across two independently-evolving repos.

**Resolution:** hash the canonicalized **raw JSON-RPC `arguments`** minus declared control fields — **not** the Zod-parsed output. Rationale: governance mints the token from the exact args it puts on the wire; the only shape both repos reproduce deterministically _without sharing connector Zod schemas_ is the wire shape. Hashing post-parse output would inject Zod defaults/coercions/transforms/unknown-key stripping/omitted-optionals → false `ACTION_HASH_MISMATCH`.

- Implemented as exported `canonicalizeExecutableArgs({ rawArgs, exclude })`.
- Each governed write declares its control fields via new annotation field **`executableArgsExclude: string[]`** (e.g. `["dry_run"]`).
- Pinned by golden vectors; flagged as the one item requiring explicit governance confirmation (does governance hash wire args minus the same exclude set?).

**Open item for governance:** confirm wire-args-minus-exclude is the shared shape. If governance instead hashes post-defaulted args, we converge on a shared canonicalization both call — but wire args is the recommended decoupled default.

## 6. Header threading

- Add `x-cesteral-decision-token` to every server's `corsAllowHeaders`.
- `mcp-http-transport-factory` extracts the header and sets typed **`RequestContext.decisionToken?: string`** (not index-signature). The factory already runs inside `runWithRequestContext`; the tool ALS context preserves parent fields, so the value reaches the handler unchanged.
- **stdio:** no header path. Token absent → `warn` logs `MISSING_TOKEN`; `enforce` blocks. stdio deployments that want writes run mode `off` or supply tokens out-of-band (matches self-host posture).

## 7. Contract-surface layer — `writeClass` migration

Treated as a **real schema migration in PR 1**, even though all current tools stay `entity`, so governance can adapt against a stable contract before the 122-tool rollout.

```ts
// cesteral write annotation gains:
writeClass: "entity" | "effect"
executableArgsExclude: string[]            // control fields excluded from actionHash

// Conditional on writeClass (no longer unconditionally required):
readPartner?:          required for "entity", omitted for "effect"
requiresValidation:    true for "entity"; boolean for "effect"
requiresSimulation:    true for "entity"; boolean for "effect"
supportsBeforeAfterSnapshot: true for "entity"; false for "effect"
```

### entity writes (existing 13 + new create/delete/bulk_update_status on canonical kinds)

Unchanged contract: `before`/`after` `NormalizedEntitySnapshot`, native-or-symbolic dry-run, `dispatchedCapability.canonicalEntityKind` non-null.

### effect writes (uploads, report-schedule CRUD, conversion uploads, graphql bulk, adjust-bids w/o canonical snapshot)

```ts
structuredContent.effect = {
  effectKind: "asset_created" | "schedule_created" | "schedule_deleted"
            | "conversions_uploaded" | "bulk_job_submitted" | ...,
  summary: Record<string, scalar>,   // idempotency/audit identity only — see redaction
}
structuredContent.dispatchedCapability = { operation, canonicalEntityKind: null }
```

- `DispatchedCapabilitySchema.canonicalEntityKind` becomes **nullable**.
- Dry-run: symbolic-validate-only where feasible (`wouldSucceed` + symbolic `expectedEffect`); honestly `supportsDryRun: false` where not. `assertGovernedEffectDryRun` parallels `assertGovernedDryRunResult` but never throws post-validation in execute mode.

### effect.summary redaction (day one)

`summary` carries only idempotency/audit identity (asset id, schedule id, accepted/rejected counts). Raw uploaded payloads / conversion PII redacted via the existing `http-request-recorder` redaction conventions. Upload/conversion tools get redaction rules from the first PR that annotates them.

### Downstream `cesteral-intelligence` changes (out of this repo, gated behind warn-only)

Analogous to the entity-kind registration playbook: accept `writeClass`, nullable `canonicalEntityKind`, the `effect`/`DryRunResult` effect variant, new operation enums, `executableArgsExclude`. Enumerated as a per-batch downstream checklist.

## 8. JtiStore

```ts
interface JtiStore {
  consumeOnce(jti: string, ttlMs: number): Promise<"fresh" | "replayed">;
}
```

- `InMemoryJtiStore` — default for stdio/self-host/single-instance/CI. TTL ≥ token TTL.
- `FirestoreJtiStore` — atomic create-if-absent; **hosted default when enforcement is enabled** (in-memory is not meaningful across Cloud Run scale-out).
- Selected by env (like `INTERACTION_LOG_MODE`). **Warn loudly if `enforce` resolves with an in-memory store** on a multi-instance deployment.

## 9. Error handling & telemetry

- `warn`-mode failures never break execution; `enforce` → 401 via `McpError(Unauthorized)`.
- Structured Pino audit logs (`component: "governance-audit"`) for **all** verdict states: `MISSING_TOKEN`, `MALFORMED_TOKEN`, `UNSUPPORTED_ALG`, `INVALID_SIGNATURE`, `MISSING_CLAIM`, `EXPIRED`, `WRONG_ISSUER`/`WRONG_AUDIENCE`, `CONTRACT_MISMATCH`, `DEFINITION_HASH_MISMATCH`, `ACTION_HASH_MISMATCH`, `REPLAYED_JTI`, `OK`. Fields: `sub` (tenant), `contractId`, `toolName`, `jti`, `status`, `reasonCode`, `detail` (e.g. missing claim name), `mode`. **Never log raw token or secret.**
- Metric: `recordDecisionTokenVerification(reasonCode, mode)` counter alongside existing OTEL metrics.

## 12. Open items requiring governance confirmation

**Verified against `cesteral-intelligence` on `main` (2026-06-02) — NOT yet compatible.** Governance computes `actionHash = sha256(stableStringify(stripInternalExecutionArgs(args)))`:

- `lib/features/governance/decisions/mutations.ts:33` (`hashActionInput`)
- `lib/features/governance/utils.ts:19` (`stableStringify`, manual recursive sort)
- `lib/features/mcp/tools/external-governance.ts:960` (`stripInternalExecutionArgs`, strips `__*` only)

For valid wire JSON the byte output is identical to the connector's `JSON.stringify(sortKeysDeep)` (both sort with `Object.keys().sort()`; verified `utils.ts:30` vs `contract-hash/src/index.ts:34`). The actionable gap is that governance strips `__*` but not `executableArgsExclude`.

**Verified transmit==hash invariant.** `stripInternalExecutionArgs(args)` at `external-governance.ts:165` produces the `executableArgs` that governance both **transmits** to the connector (`executeMcpTool(..., executableArgs)`, `external-governance.ts:645-646`) **and** hashes into the token (`hashActionInput(executableArgs)`, `external-governance.ts:858`). Two consequences:

- `__*` is stripped **upstream**, so it never reaches the wire the connector sees — the connector's own `__*` strip is an idempotent no-op for governance-originated traffic (kept for defense-in-depth / non-governance callers, not for parity).
- Governance's `actionHash` is over **exactly the bytes it sends**. The connector re-hashes those same bytes after additionally removing `executableArgsExclude`. Therefore the **sole** residual divergence is any `executableArgsExclude` field (e.g. `dry_run`) present in the transmitted args.

See the implementation-plan Risks table for the full matrix.

1. **Golden vectors** for `stableStringify` generated by governance — until then connector vectors are `pending`. Target: governance imports `@cesteral/contract-hash` so there is one implementation.
2. **"Executable args" shape** = canonicalized **wire args minus `__*` minus `executableArgsExclude`**. Connector strips both today; governance strips only `__*`. Because governance hashes the exact bytes it transmits (verified `external-governance.ts:165→645/858`, see above), `__*` parity is moot (already removed upstream, never on the wire) and the **single** residual divergence is `executableArgsExclude`: governance leaves those fields in the hashed/transmitted args, the connector strips them. Governance must extend stripping to `executableArgsExclude` per contract before any `enforce` flip — §5.
3. Governance acceptance of the `writeClass`/`effect`/nullable-`canonicalEntityKind` contract shape before effect-tool admission.

**No contract is flipped to `enforce` until items 1 + 2 are resolved on the governance side.**
